# agent-memory improvement plan

Synthesized from full codebase review + research on ReMe, Letta, MemoryOS, MaRS, LangMem, and Park et al.

---

## Executive summary

The repo has a solid foundation: clean `Store` interface, FTS5+vector hybrid search, budget-aware context assembly, versioning, TTL/GC, links, and file refs — all in pure Go SQLite. Three things are needed to make it genuinely effective as an agent memory backend:

1. **Fix retrieval quality** — two scoring bugs cause recency and relevance to be nearly ignored right now
2. **Add the reflect loop** — the only major structural gap relative to the research; required for memory quality to improve over time
3. **Add mem_peek + tier-awareness** — the lazy discovery + injection budget machinery from the architecture design

Everything is structured as sequenced Claude Code tasks respecting CONVENTIONS.md.

## Current state audit

### What's solid

- `Store` interface is well-designed and extensible without breakage
- Memory model has the right shape: NS/Key/Kind/Tags/Priority/AccessCount/LastAccessedAt/ExpiresAt/Versioning
- Three-tier search pipeline: FTS5 → LIKE fallback → vector cosine
- Context assembly with greedy budget packing
- Links (relates_to, contradicts, depends_on, refines), file refs, GC, TTL, chunking
- Pure Go SQLite (modernc.org/sqlite), WAL mode, no CGo

### Bugs in the current scoring formulas

**Bug 1 — Recency formula in `search.go` SQL is broken:**

```sql
-- CURRENT (broken):
(julianday(m.created_at) / julianday('now')) * 0.3
-- julianday('now') ≈ 2,460,000. A 7-day-old memory has
-- julianday ≈ 2,459,993. Ratio ≈ 0.9999971. Effectively 1.0 for all memories.
-- Recency has ZERO discriminating power.

-- CORRECT:
math.Exp(-0.1 * age_in_days)
-- Or in SQL: (1.0 / (1.0 + (julianday('now') - julianday(m.created_at))))
```

**Bug 2 — `context.go` ignores similarity scores from search:**

```go
// CURRENT (context.go line ~60):
relevance := 1.0  // all memories get the same relevance regardless of FTS rank or vector similarity

// CORRECT: use the SearchResult.Similarity field which is already computed
relevance := 1.0 // base
if r.Similarity > 0 {
    relevance = r.Similarity  // actually use the vector score
}
```

**Bug 3 — `access_count` / `last_accessed_at` not updated on `Search()` or `Context()`:**

Only `Get()` updates these fields. Search and context calls don't register as access events, so the access-frequency component of scoring is always stale.

**Bug 4 — Vector search does a full table scan:**

`searchVector()` fetches ALL chunks with embeddings into memory and computes cosine similarity in-process. For 10K+ memories this becomes slow. Needs a hard candidate limit or sqlite-vec ANN index.

## Improvement phases

### Phase 1 — Fix retrieval scoring (no schema changes, no new deps)

**Goal:** Make the existing scoring actually work. High impact, zero risk of breaking changes.

**Task 1.1 — Fix recency decay in `search.go` SQL query**

File: `internal/store/search.go`

Replace the broken `julianday` ratio with an actual decay formula:

```sql
-- Replace:
(julianday(m.created_at) / julianday('now')) * 0.3

-- With (exponential decay, half-life ~7 days):
(1.0 / (1.0 + (julianday('now') - julianday(m.created_at)) / 7.0)) * 0.3
```

This is a pure SQL change, no schema touch, fully backwards compatible.

**Task 1.2 — Use similarity scores in `context.go`**

File: `internal/store/context.go`

The `SearchResult` struct already carries `Similarity float64`. Pass it through to the scoring:

```go
// In Context(), when iterating results:
relevance := 0.5 // base for FTS matches
if r.Similarity > 0 {
    relevance = r.Similarity // use actual cosine similarity when available
}
score := relevance*0.4 + recency*0.2 + importance*0.2 + accessFreq*0.2
```

**Task 1.3 — Update access metadata on Search and Context calls**

File: `internal/store/sqlite.go` (add helper), `internal/store/search.go`, `internal/store/context.go`

After assembling results in `Context()`, batch-update `access_count` and `last_accessed_at` for all returned memory IDs. This makes the access-frequency score meaningful.

```go
func (s *SQLiteStore) touchMemories(ctx context.Context, ids []string) error {
    if len(ids) == 0 {
        return nil
    }
    placeholders := strings.Repeat("?,", len(ids))
    placeholders = placeholders[:len(placeholders)-1]
    now := time.Now().UTC().Format(time.RFC3339)
    args := []interface{}{now, now}
    for _, id := range ids {
        args = append(args, id)
    }
    _, err := s.db.ExecContext(ctx,
        fmt.Sprintf(`UPDATE memories SET access_count = access_count + 1, last_accessed_at = ? WHERE id IN (%s) AND deleted_at IS NULL`, placeholders),
        args...,
    )
    return err
}
```

**Task 1.4 — Cap vector search candidate set**

File: `internal/store/search.go`

Add a candidate limit to `searchVector()` — fetch the 500 most-recent memories with embeddings rather than all of them. For now this is a pragmatic cap; ANN indexing via sqlite-vec is a later phase.

```go
// In searchVector SQL, add:
ORDER BY m.created_at DESC LIMIT 500
```

### Phase 2 — Schema additions for richer scoring

**Requires human approval per CONVENTIONS.md (schema changes).**

**New fields to add via `ALTER TABLE`:**

```sql
-- Add to memories table:
ALTER TABLE memories ADD COLUMN importance REAL NOT NULL DEFAULT 0.5;
-- 0.0-1.0 continuous score, set by caller at write time (not LLM-computed)
-- Replaces the coarse 4-level priority as the primary importance signal
-- priority field kept for backwards compatibility

ALTER TABLE memories ADD COLUMN utility_count INTEGER NOT NULL DEFAULT 0;
-- Incremented by caller when a retrieved memory actually contributed to success
-- Enables ReMe-style u(E)/f(E) utility ratio for pruning

ALTER TABLE memories ADD COLUMN tier TEXT NOT NULL DEFAULT 'stm';
-- 'stm' | 'ltm' | 'dormant' | 'archived' | 'identity'
-- Set by reflect process, used for injection priority

ALTER TABLE memories ADD COLUMN est_tokens INTEGER NOT NULL DEFAULT 0;
-- Pre-computed at write time: len(content)/4 + 20
-- Enables budget-accurate context assembly without per-query estimation
```

File: `internal/store/sqlite.go`

Add to `migrate()` using `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` pattern (SQLite supports this since 3.37). All new columns have safe defaults so existing data remains valid.

**Task 2.2 — Pre-compute `est_tokens` at write time**

File: `internal/store/sqlite.go` (in `Put()` path)


**Task 2.3 — Update `ContextParams` to use `est_tokens` for budget packing**

File: `internal/store/context.go`

Switch from `len(c.memory.Content)` to `c.memory.EstTokens` in the greedy packing loop. Makes budget accounting accurate and avoids truncating memories mid-sentence.

### Phase 3 — Reflect loop

**The headline missing feature. Requires human approval (new table, Store interface change, new CLI subcommand).**

**Architecture:** Rule-based, no LLM inside the library. Rules are condition→action pairs stored in a new `reflect_rules` table. The reflect engine evaluates rules against batches of memories and applies ADD/UPDATE/DELETE/DECAY/PROMOTE/ARCHIVE operations.

**Task 3.1 — Add `reflect_rules` table**

```sql
CREATE TABLE IF NOT EXISTS reflect_rules (
    id          TEXT PRIMARY KEY,
    ns          TEXT NOT NULL DEFAULT '',      -- '' = applies to all namespaces
    name        TEXT NOT NULL,
    priority    INTEGER NOT NULL DEFAULT 50,
    scope       TEXT NOT NULL DEFAULT 'reflect', -- 'retain' | 'reflect' | 'injection' | 'all'
    created_by  TEXT NOT NULL DEFAULT 'system', -- 'system' | 'agent:<id>'
    -- Condition fields (all optional, AND-joined when present)
    cond_tier           TEXT,       -- match memories in this tier
    cond_age_gt_hours   REAL,       -- older than N hours
    cond_importance_lt  REAL,       -- importance below threshold
    cond_access_lt      INTEGER,    -- fewer than N accesses
    cond_utility_lt     REAL,       -- u/f ratio below threshold
    cond_kind           TEXT,       -- match memory kind
    cond_tag_includes   TEXT,       -- memory has this tag
    -- Action fields
    action_op           TEXT NOT NULL, -- DECAY | DELETE | PROMOTE | DEMOTE | ARCHIVE | TTL_SET | PIN
    action_params       TEXT,          -- JSON params for op
    -- Rule TTL
    rule_expires_at     TEXT,          -- NULL = permanent
    created_at          TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_rules_ns ON reflect_rules(ns, scope);
CREATE INDEX IF NOT EXISTS idx_rules_priority ON reflect_rules(priority DESC);
```

**Task 3.2 — Add reflect engine to store package**

File: `internal/store/reflect.go` (new file)

```go
// ReflectParams controls a reflect cycle.
type ReflectParams struct {
    NS          string
    DryRun      bool
    ImportanceThreshold float64  // sum of recent importance triggers reflect (default: 20.0)
    CountThreshold      int      // new memories since last reflect triggers it (default: 20)
}

// ReflectResult summarizes what the reflect cycle did.
type ReflectResult struct {
    MemoriesEvaluated int
    RulesApplied      int
    Decayed           int
    Promoted          int
    Demoted           int
    Archived          int
    Deleted           int
    Errors            []string
}
```

Add to Store interface:
```go
Reflect(ctx context.Context, p ReflectParams) (*ReflectResult, error)
RuleSet(ctx context.Context, rule ReflectRule) (*ReflectRule, error)
RuleGet(ctx context.Context, id string) (*ReflectRule, error)
RuleList(ctx context.Context, ns string) ([]ReflectRule, error)
RuleDelete(ctx context.Context, id string) error
```

**Built-in system rules (loaded at DB init):**

```go
var builtinRules = []ReflectRule{
    {
        ID: "sys-decay-unaccessed",
        Name: "Decay importance for unaccessed STM memories",
        Scope: "reflect",
        Priority: 10,  // lowest, runs last
        CreatedBy: "system",
        Cond: RuleCond{Tier: "stm", AgeGTHours: 72, AccessLT: 3},
        Action: RuleAction{Op: "DECAY", Params: map[string]any{"factor": 0.95, "min": 0.1}},
    },
    {
        ID: "sys-promote-to-ltm",
        Name: "Promote accessed STM to LTM",
        Scope: "reflect",
        Priority: 50,
        CreatedBy: "system",
        Cond: RuleCond{Tier: "stm", AccessGT: 3, AgeGTHours: 24},
        Action: RuleAction{Op: "PROMOTE", Params: map[string]any{"to_tier": "ltm"}},
    },
    {
        ID: "sys-demote-stale-ltm",
        Name: "Demote LTM not accessed in 7 days",
        Scope: "reflect",
        Priority: 50,
        CreatedBy: "system",
        Cond: RuleCond{Tier: "ltm", AgeGTHours: 168, AccessLT: 2},
        Action: RuleAction{Op: "DEMOTE", Params: map[string]any{"to_tier": "dormant"}},
    },
    {
        ID: "sys-prune-low-utility",
        Name: "Delete low-utility memories",
        Scope: "reflect",
        Priority: 90,  // runs near last
        CreatedBy: "system",
        Cond: RuleCond{AccessGT: 5, UtilityLT: 0.2},
        Action: RuleAction{Op: "DELETE"},
    },
}
```

**Conflict resolution:** PRESERVE/PIN actions always beat DECAY/DELETE/DEMOTE at equal priority. Highest numeric priority wins otherwise. If two rules conflict, log the conflict in `ReflectResult.Errors` — don't silently discard.

### Phase 4 — `mem_peek` and injection manifest

**Goal:** Enable the lazy discovery pattern. Agent can call `mem_peek` to get a lightweight index of what memory exists before deciding what to pull.

**Task 4.1 — Add `Peek` to Store interface**

File: `internal/store/store.go`

```go
// PeekResult is a lightweight memory index for lazy discovery.
type PeekResult struct {
    NS              string            `json:"ns"`
    IdentitySummary string            `json:"identity_summary,omitempty"` // first identity-tier memory content
    RecentTopics    []string          `json:"recent_topics"`              // top-5 tags by recency
    MemoryCounts    map[string]int    `json:"memory_counts"`              // by tier
    HighImportance  []MemoryStub      `json:"high_importance"`            // top-5 by importance score
    TotalEstTokens  map[string]int    `json:"total_est_tokens"`           // tokens per tier
}

type MemoryStub struct {
    ID         string  `json:"id"`
    Key        string  `json:"key"`
    Kind       string  `json:"kind"`
    Tier       string  `json:"tier"`
    Importance float64 `json:"importance"`
    EstTokens  int     `json:"est_tokens"`
    Summary    string  `json:"summary"` // first 80 chars of content
}

// Peek returns a lightweight index of memory state for lazy discovery.
// Costs: 2 SQL queries, returns ~200 tokens of data.
Peek(ctx context.Context, ns string) (*PeekResult, error)
```

**Task 4.2 — Add `peek` CLI subcommand**

```
agent-memory peek [--ns <ns>]
```

Returns JSON matching `PeekResult`. Agent uses this to orient itself at session start without loading full memory content.

**Task 4.3 — Extend `ContextParams` with tier-aware injection**

File: `internal/store/context.go`

```go
type ContextParams struct {
    NS           string
    Query        string
    Kind         string
    Tags         []string
    Budget       int    // max tokens
    // New fields:
    PinTiers     []string  // tiers always injected (e.g. ["identity", "ltm"])
                           // loaded first, before query-relevant search
    PinBudget    int       // token budget reserved for pinned tiers (default: Budget/3)
    SearchBudget int       // remaining budget for query-relevant search (default: Budget - PinBudget)
}
```

The new context assembly flow:
1. Load all memories with `tier IN (PinTiers)` ordered by importance DESC, fill `PinBudget`
2. Run hybrid search for query-relevant memories, fill `SearchBudget`
3. Merge, dedup, return

This is the eager (pinned) + lazy-capable (query-search) split in a single `Context()` call.

### Phase 5 — Hybrid search fusion (longer term)

Currently search is sequential: FTS5 results → supplement with LIKE → supplement with vector. This is additive, not fused. True hybrid search merges all three signals with weighted scoring before returning.

**Task 5.1 — Implement RRF (Reciprocal Rank Fusion)**

File: `internal/store/search.go`

```go
// RRF score: sum of 1/(k + rank_i) for each retrieval method
// Standard k=60 per the original RRF paper
func rrfScore(ranks []int, k int) float64 {
    var score float64
    for _, r := range ranks {
        score += 1.0 / float64(k+r)
    }
    return score
}
```

Run FTS5, LIKE, and vector searches independently (parallel goroutines), then merge by RRF score. This is the approach both ReMe (0.7 vector + 0.3 BM25) and OpenClaw use. The weighting is configurable via `SearchParams`.

**Task 5.2 — Consider sqlite-vec for ANN**

The current vector search is O(n) in memory count. For OpenClaw's use case (one agent, hundreds of memories) this is fine. If the store is shared across multiple agents or grows past ~5K memories, investigate `sqlite-vec` for approximate nearest neighbor. No action needed until benchmarks show latency >100ms.

## Key design decisions to confirm before Phase 3

1. **Reflect trigger:** Should reflect run on a timer (background goroutine in the binary) or only when `agent-memory reflect` is called explicitly? Given the tool-first design philosophy, recommend CLI-only trigger with MCP server calling it on a schedule. Avoids background goroutines in a CLI tool.

2. **Importance score at write time:** Who sets `importance`? Recommend: caller passes it in `PutParams`. The `--importance` flag on `agent-memory put` lets the agent declare importance explicitly. No LLM call inside the library. Default 0.5 for backwards compatibility.

3. **Rules persistence:** Should built-in system rules be re-seeded on every startup (idempotent), or stored once and user-editable? Recommend: seed on startup with `ON CONFLICT IGNORE` — built-ins are always present but user rules with the same ID override them.

4. **Utility tracking:** The `UtilityInc` tool must be called by the agent after successful task completion. This requires agent discipline. Alternative: infer utility from access patterns (if a memory was retrieved and the conversation continued successfully). Recommend: explicit `UtilityInc` for now, observe if agents actually call it in practice.
